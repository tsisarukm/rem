# Design-Doc: LLM-based project — Исторический RAG-агент "Хронорус"

## 1. Контекст проекта
### 1.1 Бизнес-задача
- **Описание проблемы:** При изучении истории (в частности России) или подготовке исторических справок крайне важна фактическая точность, особенно касательно дат и последовательности событий. Обычные LLM склонны "галлюцинировать", выдумывая даты, путая эпохи или просто не находя ответ на сложный вопрос.
- **Решение:** Создание RAG-системы, которая строго опирается на базу знаний (учебники, энциклопедии, вики) и использует гибридный (или каскадный) поиск для точного извлечения дат и фактов. Все на русском языке, ну по понятным причинам.
- **Критерии успеха:** 
  1. Бот не придумывает несуществующие даты (процент галлюцинаций с датами < некоторого порога).
  2. Бот честно отказывается отвечать, если факта нет в базе.
  3. Бот реально хорошо отвечает на историчесике вопросы (полнота большая)

### 1.2 Целевая аудитория и пользователи
- Обычные внешние пользователи (скорее школьники - поэтому вопрос не по теме без ответа (см пункты ниже))
- Сценарий: текстовый чат через веб-интерфейс. Нагрузка небольшая (локалхост, наверное 5-10 рпс? Зависит от от OpenRouter)

### 1.3 Ограничения и допущения
- Система отвечает **только** на вопросы по истории России. На вопросы из других областей бот выдает вежливый отказ.
- Мы не разворачиваем (по крайней мере не планируем) тяжелую LLM локально из-за отсутствия стабильного числа GPU. Используется внешнее бесплатное API (OpenRouter).
- Данные статичны, обновление базы "на лету" не требуется.

## 2. Архитектура решения
### 2.1 Общая схема
На этапе MVP и учебного тестирования **практически вся архитектура развернута локально (на компьютере разработчика)**. Исключение составляет только генеративная LLM, к которой мы обращаемся по сети.
Схема такая:

`[Пользователь]` 

   ↓ (текстовый запрос)

`[1. Фронтенд (Streamlit)]` --- (работает локально)

   ↓ (HTTP POST request)

`[2. API Бэкенд (FastAPI)]` --- (работает локально)

   ↓ 

`[3. Эмбеддер]` --- (Локальная модель sergeyzh/BERTA на CPU)

   ↓ (вектор запроса + извлеченные из вопроса эвристики/даты)

`[4. База знаний (Qdrant)]` --- (лежит локально в Docker-контейнере)

   ↓ (возвращает Топ-K релевантных текстовых чанков)

`[5. Формирование Промпта]` --- (захардкожено в бэкенде)

   ↓ (Системный промпт + Контекст из Qdrant + Вопрос юзера)

`[6. Внешняя LLM]`          --- (OpenRouter API (Интернет))

   ↓ (сгенерированный ответ)

`[7. Пост-обработка]`       --- (FastAPI: проверка на отказы и форматирование)

   ↓

`[Вывод ответа в Streamlit UI]`


### 2.2 Хранилище знаний
- **Источники:** Статьи Википедии (категория "История России"), учебники истории, энциклопедии 
- **Предобработка:** Парсинг текста, разбиение на чанки (по абзацам, видимо с оверлэпом, надо еще поместиться в контекст моделкьи - в базовом сценарии это 512 токенов). Применение регулярных выражений для выделения годов в чанке и добавления их в метаданные. Возможно еще извлечение метаданных типо имен, географичеких мест, в общем все такие "эвристические" данные для ускорения и точности.
- **Обновление:** Ручной скрипт индексации, запускается один раз оффлайн. Если понадобится все же обновить, то ну запустится еще раз.

### 2.3 Интеграции и интерфейсы
- **API:** RESTful API (FastAPI)
- **UI:** Простой веб-чат на Streamlit (или Gradio) с отображением истории сообщений (мб + панель "Использованные источники").

### 2.4 Инфраструктура и развертывание
- **Стек:** Python, FastAPI, Docker, Docker Compose, Qdrant, OpenRouter, SentenceTransformer, Hugging Face.
- **Деплой:** Локальный запуск через `docker-compose up` (поднимает контейнер с Qdrant, бэкендом и фронтом). Вычисления (эмбеддинги) на CPU.

## 3. Данные и качество знаний
### 3.1 Сбор и предобработка данных
- Формат: TXT и Markdown.
- Выделение сущностей: при чанковании в метаданные Qdrant добавляем поле `mentioned_years` (список дат, найденных в тексте) и по аналогии `mentioned_places`, `mentioned_people` (тут надо подумать - хочется очень важных людей, которые часто встречаются). Возможны еще доработки.

### 3.2 Векторизация и индексирование
- **Модель embedding:** `sergeyzh/BERTA` в идеале. Одна из топовых (на момент написания этого файлика) в лидерборде HF русскоязчных retrieval моделек, при этом самая легкая из тех, что обладают адекватным Retireval. Будем ли мы пробовать потяжлее для выбивания профита (как минмум есть FRIDA - потяжелее, но не сильно, но она вообще в ТОП5) или наоборот легче (не хочется, ибо большая потеря качества), узнаем по мере выполнения. Также как и мб 512 токенов нам захочется увеличть - тогда есть чуть потяжелее модель - `USER-bge-m3`
- **Параметры индекса:** Косинусное расстоянре. Планируем использование механизма Qdrant Sparse Vectors для гибридного поиска, возможны свои наработки, типо каскадного поиска (надо изучать возможности Qdrant).

### 3.3 Метрики качества знаний
- **Покрытие:** Контрольный пул из 30-50 вопросов из ЕГЭ по истории.
- **Проверка:** Оценка Retrieval-части с помощью метрик ранжирования (Hit@, DCG,..) на контрольных вопросах. В целом сами спокойно можем накдиать вопросов, как люди, понимающие что-то в истории России и побыть ассесорами.

## 4. Модель и генерация
### 4.1 Выбор LLM и промптинг
- **Модель:** Одна из мощных бесплатных моделей в OpenRouter (например, `Qwen: Qwen3 Coder 480B A35B (free)`,`Meta: Llama 3.3 70B Instruct (free)` или `Google: Gemma 3 27B (free)`). Надо посмотреть на время ответа моделек и стабильность, ну и качество, фаворит - квен, я и даже по работе пользуюсь.
- **Промпт-шаблоны:** Базово что-то типо: "Ты — строгий учитель истории. Используй ТОЛЬКО следующий контекст для ответа. Если ответа нет в тексте, напиши 'У меня нет достоверной информации об этом факте.' Контекст: {context}". Неплохо было бы добавить пару защит от тривиальных промпт-инъекций и точно убедится, что модель не отвечает "плохие" вопросы - нащупаем.

### 4.2 Контроль качества ответов
- Ручная визуальная проверка (Human Evaluation) на тестовых примерах с датами. 
- Контроль галлюцинаций: проверка того, что все даты (и тд.) в ответе присутствуют в исходных данных.

## 5. UX / пользовательский опыт
### 5.1 Сценарии взаимодействия
- Приветствие: "Привет! Я Хронорус. Задайте мне любой вопрос по истории России."
- Плохие вопросы: на вопрос "Как сварить борщ?" ответ "Я специализируюсь только на истории России, пожалуйста, задайте другой вопрос."
- Вопросы по истории России, но на которые нет ответа в данных: "Прошу прощения, я не знаю ответ на этот вопрос"

### 5.2 Диалоговая логика
- Single-turn логика (отдельные вопросы-ответы без глубокой памяти, так как исторические факты чаще всего запрашиваются изолированно). В UI возможно покажем ссылки на исходные абзацы текста, откуда взят ответ (это мб сложно).

## 6. Безопасность, соответствие и этика
1. Сама локальная система (бэкенд) не собирает и не логирует никакие данные. **ОДНАКО**, поскольку используется внешнее бесплатное API (OpenRouter), все пользовательские запросы и найденный контекст передаются на сторонние сервера. Согласно ToS большинства бесплатных моделей, эти данные могут логироваться и использоваться для дообучения. Этой проблемы не будет если использовать развернутые нейронки, но в первой итерации мы точно будем использовать внешние.
2. В UI чата будет добавлено явное предупреждение: *"Пожалуйста, не вводите свои персональные данные, пароли и чувствительную информацию"*.
3. Возможно добавление функционала-защиты от утечки данных все же. Например, LLM as a judge (развернуть все же но очень маленькую модель и ей проверять нет ли утечек) или использовать маскирование текста умное.
4. Бот должен быть этичным и вообще не говорить на инные темы кроме истории Росиии

## 7.  План внедрения и эксплуатации
### 7.1  Этапы проекта
1. Сбор исторических текстов и настройка скрипта чанкования (добавление метаданных-дат).
2. Разворачивание Qdrant через Docker, создание функции гибридного поиска.
3. Написание FastAPI + подключение OpenRouter API. Формирование промптов.
4. Обертка в Streamlit, сборка итогового `docker-compose` и защита проекта.

### 7.2 Поддержка и эксплуатация
1. Точного разделения между нами троими нет - отвечаем за работу сервера все вместе.

## 8. Риски и допущения
1. Модель ошибается, хотя у нее есть данные для ответа.
2. Модель все же галлюцинирует, если у нее нет данных для ответа.
3. Модель нарушает пункт 6 (про безопасность и этику), особенно опасен слив данных.

## 9. Бюджет и ресурсы
1. Три ML разработчика
2. Бесплатные GPU (kaggle/collab) и небольшая возможность использовать рабочие A800
3. Личные CPU
